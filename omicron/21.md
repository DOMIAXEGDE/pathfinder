# Prompt
```
Write a complete, compilable, standard C++17 program named 16.cpp.

The program is a deterministic basis-tensor generator. Its purpose is to convert six user-provided text strings per instance into a six-coordinate raw-id tensor and a six-coordinate least-residue tensor. After N instances, it outputs parse-observable tensor datasets of shape [N, 6]. The output is intended to be consumed by later instruction sets, simulations, thought-experiment frameworks, or formal computation engines, but 16.cpp itself must not claim to perform physics simulation, solve arbitrary problems, prove arbitrary theorems, or generate uncountable objects exactly.

Use the mathematical scalar form from Structure 1:

a_i = (p / q) * ((m / g) ^ (alpha / beta)).

Use alpha instead of a for the exponent numerator in code and documentation, because a_i and A already use the letter a. Let A denote the exact scalar family generated by this expression only when the expression is mathematically defined in the selected scalar domain. The variables p, m, and alpha are integer slots. The variables q, g, and beta are positive-integer slots.

The six tensor coordinates are not arbitrary hashes. Each coordinate is a deterministic integer id produced by a match-paste encoding function applied to one pasted text string, one finite ordered alphabet, and one length policy. The six components are ordered exactly as:

p, q, m, g, alpha, beta.

For each component c in {p, q, m, g, alpha, beta}, the user supplies a finite ordered alphabet Sigma_c. The alphabet must contain no duplicate decoded symbols. For the minimum implementation, symbols may be treated as bytes after JSON or interactive escaped input is decoded. The program must document whether it is operating in byte-symbol mode or a richer character-symbol mode. In byte-symbol mode, the alphabet is the ordered sequence of decoded bytes in the alphabet string, and input lengths are byte lengths. The program must reject any input text containing a byte or symbol not in the component alphabet.

For a fixed-length component with alphabet size r and length L, encode a text string x = x_0 x_1 ... x_{L-1} by mapping each symbol x_k to its alphabet index d_k in {0, ..., r - 1}. Use big-endian lexicographic ordinal encoding:

id(x) = sum from k = 0 to L - 1 of d_k * r^(L - 1 - k).

The raw coordinate domain size is:

M = r^L.

The id must satisfy:

0 <= id < M.

For a variable-length component with alphabet size r and allowed length interval [L_min, L_max], the actual input length L must satisfy:

L_min <= L <= L_max.

Encode the text string by first computing the fixed-length ordinal id_L(x), then adding the offset for all shorter valid lengths:

offset(L) = sum from ell = L_min to L - 1 of r^ell.

The variable-length id is:

id(x) = offset(L) + id_L(x).

The raw coordinate domain size is:

M = sum from ell = L_min to L_max of r^ell.

This rule guarantees that strings of different allowed lengths do not collide. Length values must be nonnegative integers. Fixed length L = 0 and variable minimum L_min = 0 may be supported if the implementation correctly handles the empty string; otherwise the program must clearly document that it requires positive lengths and must reject zero-length policies during validation.

Define the least nonnegative residue of an integer z modulo a positive integer mu as the unique integer rho such that 0 <= rho < mu and rho is congruent to z modulo mu. By default, reduce each raw coordinate id modulo that component's raw coordinate domain size M_c. Since the deterministic encoder already yields 0 <= id_c < M_c, the default least-residue coordinate is the raw id itself. Still implement the reduction function explicitly so that user-provided alternate moduli can be supported.

Allow the user to override the default residue modulus for any component by providing an explicit positive integer modulus mu_c. If mu_c is provided, the residue is:

rho_c = id_c mod mu_c.

If no explicit modulus is provided, the residue is:

rho_c = id_c mod M_c.

Store the original raw id, the selected residue modulus, and the least residue in the output. Since alternate moduli can collapse distinct raw ids to the same residue, the program must never treat a residue tensor as sufficient to reconstruct the original raw input unless the selected moduli make that reconstruction injective and the program explicitly proves or checks that fact.

For instance i, after processing the six component strings, form the raw id tensor:

I_i = [id_p, id_q, id_m, id_g, id_alpha, id_beta].

Then form the least-residue tensor:

T_i = [rho_p, rho_q, rho_m, rho_g, rho_alpha, rho_beta],

where rho_c is the least nonnegative residue of id_c modulo the selected component residue modulus.

After N instances, form two explicitly named datasets:

B_raw = [I_0, I_1, ..., I_{N-1}],

and

B_residue = [T_0, T_1, ..., T_{N-1}].

Both datasets have shape [N, 6]. B_raw is the reconstruction-addressed raw tensor dataset. B_residue is the displayed least-residue tensor dataset. When using the phrase basis tensor dataset without qualification, the program must either refer to both datasets together or explicitly state whether it means B_raw or B_residue.

Do not call either dataset a linear algebra basis unless the user explicitly requests an independence check and the program actually performs one. By default, call the output a basis-tensor candidate set or basis tensor dataset. If an independence check is implemented, it must state the algebraic structure used, such as row rank over a user-selected prime field F_P. Do not imply independence over the integers or over a ring of residues unless it is actually proven by the implemented check.

Map raw ids to Structure-1 scalar parameters separately from the least-residue tensor. For integer slots p, m, and alpha, use a deterministic signed mapping. The default signed mapping must be zigzag:

0 -> 0,
1 -> 1,
2 -> -1,
3 -> 2,
4 -> -2,
5 -> 3,
6 -> -3,
and so on.

Equivalently, for raw id n, if n = 0 then zigzag(n) = 0; if n is odd then zigzag(n) = (n + 1) / 2; if n is even and n > 0 then zigzag(n) = -(n / 2). This mapping may produce signed arbitrary-size integers, so the implementation must support signed integer values for mapped Structure-1 parameters.

For positive-integer denominator slots q, g, and beta, use:

positive_value = raw_id + 1.

This guarantees q >= 1, g >= 1, and beta >= 1.

For each instance, construct the symbolic scalar expression:

a_i = (P_i / Q_i) * ((M_i / G_i) ^ (Alpha_i / Beta_i)),

where P_i, Q_i, M_i, G_i, Alpha_i, and Beta_i are the mapped Structure-1 parameter values derived from the raw ids. Store this expression as symbolic text and as structured JSON fields. Do not rely on floating point as the authoritative value.

The program must reduce the exponent Alpha_i / Beta_i to lowest terms for real-admissibility analysis. Since Alpha_i may be signed and Beta_i is positive, implement sign-aware rational reduction using gcd(abs(Alpha_i), Beta_i). The rational base is M_i / G_i, where M_i may be signed and G_i is positive.

Determine whether the scalar expression is real-admissible under ordinary real arithmetic. The power subexpression is undefined if the rational base is zero and the rational exponent is negative. The case 0^0 must be reported as undefined_real_expression unless the program explicitly implements and labels a separate combinatorial convention; ordinary real arithmetic must not silently define it. A zero base with a positive rational exponent is real and gives zero. A positive rational base is real for all rational exponents. A negative rational base raised to a rational exponent is real only when the reduced rational exponent has an odd positive denominator. A negative rational base with a reduced rational exponent whose denominator is even requires complex arithmetic. Multiplication by the prefactor P_i / Q_i must not be used to hide an undefined power subexpression.

Report the scalar status as one of:

real_exact_symbolic,
real_approx_available,
complex_required,
undefined_real_expression.

Use real_exact_symbolic when the expression is real-admissible and represented exactly symbolically without a numeric approximation. Use real_approx_available only when the expression is real-admissible and the program also emits a finite numeric approximation in addition to the exact symbolic form. Use complex_required when ordinary real arithmetic is insufficient but the symbolic expression is otherwise meaningful over an appropriate complex domain. Use undefined_real_expression when the expression is undefined under the stated arithmetic rules. The minimum implementation may omit numeric approximation, but it must still emit the symbolic expression and status.

Add a deterministic seed-address layer.

16.cpp must implement a deterministic integer-seed system. This seed system is not a pseudorandom generator. It is an exact rank/unrank addressing mechanism over the finite raw tensor-row universe determined by the current component alphabets and length policies. The raw row universe does not depend on N. The basis-address space depends on the raw row universe together with the selected basis row count N.

For a fixed 17.txt configuration, let the six ordered components be:

p, q, m, g, alpha, beta.

For each component c, let M_c be the raw string-domain size produced by that component's finite alphabet and length policy. For fixed length L and alphabet size r:

M_c = r^L.

For variable length [L_min, L_max]:

M_c = sum from ell = L_min to L_max of r^ell.

Define the raw tensor-row universe U as:

U = D_p x D_q x D_m x D_g x D_alpha x D_beta,

where D_c = {0, 1, ..., M_c - 1}. Its size is:

R = M_p * M_q * M_m * M_g * M_alpha * M_beta.

Every raw six-coordinate tensor row has a unique row ordinal J in {0, ..., R - 1}. Use big-endian mixed-radix ranking:

J = (((((id_p * M_q + id_q) * M_m + id_m) * M_g + id_g) * M_alpha + id_alpha) * M_beta + id_beta).

The inverse unranking must recover the component ids by repeated quotient and remainder in reverse component order:

id_beta = J mod M_beta,
J = floor(J / M_beta),
id_alpha = J mod M_alpha,
J = floor(J / M_alpha),
id_g = J mod M_g,
J = floor(J / M_g),
id_m = J mod M_m,
J = floor(J / M_m),
id_q = J mod M_q,
J = floor(J / M_q),
id_p = J.

For each recovered raw component id, reconstruct the canonical input string using the inverse of the ordinal text encoder. For a fixed-length component, write the id in base r using exactly L digits, including leading zero digits, then map each digit back to the corresponding alphabet symbol. For a variable-length component, first find the unique length L such that:

offset(L) <= id < offset(L) + r^L,

where:

offset(L) = sum from ell = L_min to L - 1 of r^ell.

Then decode id - offset(L) as a fixed-length base-r ordinal of length L. The canonical reconstructed string must round-trip through the encoder to the same raw id.

Let N be the user-specified basis row count. By default, a basis tensor dataset for seed addressing is an ordered N-tuple of raw tensor rows, with repetition allowed. Therefore the total number of ordered raw basis tensor datasets under the current configuration is:

S = R^N.

Call S the basis-address-space size. The minimum implementation must support the basis policy ordered_with_repetition. If any other basis policy is requested and not implemented, the program must reject it clearly rather than silently changing the meaning.

A raw basis tensor dataset B_raw = [row_0, row_1, ..., row_{N-1}] has a unique basis address K in {0, ..., S - 1}. Use big-endian base-R ranking:

K = (((row_0 * R + row_1) * R + row_2) ... * R + row_{N-1}).

The inverse unranking must recover rows by repeated quotient and remainder in reverse row order:

row_{N-1} = K mod R,
K = floor(K / R),

and so on until row_0 is recovered.

The program must output a canonical integer seed sequence for every generated B_raw. The seed sequence length must be user-selectable as any integer k in {1, 2, 3, 4, 5, 6}. The default seed length is 1.

For a chosen seed length k, define the seed radix W_k as the least positive integer W such that:

W^k >= S.

Equivalently:

W_k = ceil(S^(1/k)).

The program must compute W_k exactly with integer arithmetic. Floating point may be used only for hints or diagnostics, never as the authoritative result.

The canonical k-integer seed sequence for basis address K is the base-W_k expansion of K, padded with leading zeroes to exactly k integers:

seed = [s_0, s_1, ..., s_{k-1}],

where each s_j satisfies 0 <= s_j < W_k and:

K = s_0 * W_k^(k - 1) + s_1 * W_k^(k - 2) + ... + s_{k-2} * W_k + s_{k-1}.

This rule guarantees that every possible ordered raw basis tensor dataset generated by 16.cpp has exactly one canonical seed sequence of length k for each k in {1, 2, 3, 4, 5, 6}. It does not mean that every syntactically well-formed length-k digit sequence is valid in strict mode, because W_k^k can be larger than S. In strict mode, a seed sequence is valid only if every digit is less than W_k and the decoded address is less than S. In wrapping mode, out-of-range decoded addresses may be reduced modulo S as described below.

16.cpp must support an integer-seed-sequence input file named 19.txt by default. The path must be overrideable with:

--seed-file <path>

19.txt contains one seed sequence per non-empty, non-comment line. Each seed sequence must contain between 1 and 6 nonnegative decimal integers. The integers may be separated by spaces or commas. Optional square brackets may be accepted for convenience. Lines whose first non-whitespace character is # are comments.

Examples of valid 19.txt seed-line syntax are:

0
42
0 5
1, 2, 3
[0, 0, 0, 0, 0, 17]

Syntax validity is separate from strict semantic validity. For example, a syntactically valid seed may still decode to K_seed >= S and therefore be rejected in strict mode.

When reading a seed sequence from 19.txt, let k be the number of integers on that line. Compute W_k = ceil(S^(1/k)) using the current 17.txt configuration and the current value of N. Decode the seed sequence as:

K_seed = s_0 * W_k^(k - 1) + s_1 * W_k^(k - 2) + ... + s_{k-2} * W_k + s_{k-1}.

In strict seed mode, every seed digit must satisfy 0 <= s_j < W_k, and K_seed must satisfy 0 <= K_seed < S. If either condition fails, reject the seed record and report a clear validation error naming the line number in 19.txt.

In wrapping seed mode, require every seed integer to be nonnegative, decode K_seed using the same polynomial expression, and then reduce the decoded address by:

K = K_seed mod S.

The output must clearly report that wrapping occurred and must show both K_seed and the effective reduced address K. Strict seed mode must be the default because it preserves exact address semantics.
```
